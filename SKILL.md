# Knowledge Capture Skill

将对话提炼为结构化知识资产，输出可直接投入知识库的 Markdown 文档。

---

## 工作流程

### Step 1：询问整理范围

触发后只问一个问题：

```
这段对话要整理哪个范围？
A. 全部（整段对话，我来过滤噪音）
B. 指定范围（告诉我从哪里开始，比如「从我们讨论架构那里算起」）
```

- 用户选 **A** → 扫描整段对话，进入 Step 2
- 用户选 **B** → 按用户指定的起点往后，进入 Step 2
- 标题、标签、内容类型全部自动判断，不额外询问用户

---

### Step 2：过滤「批量任务」噪音

对话中可能混有大量批量生成的内容（如 100 篇文案、批量代码等），这些不是知识资产，直接跳过。

**识别信号**：
- 高度重复的结构（「第1篇」「第2篇」… 或连续相似格式）
- 明显是在执行指令而非讨论、探索、决策
- 大段内容之间缺乏对话往来

**处理规则**：
- 从对话末尾往前追溯，找到最近一段有实质讨论的内容
- 遇到批量任务区域直接跳过，不提炼批量输出本身
- 如果批量任务前后都有有价值的讨论，两段都提炼，进入下一步判断是否拆分

**典型场景——架构调试中夹带批量输出**：

用户在调试某个 Agent / 工作流时，会穿插让 AI 批量生成内容来验证效果。这类对话的知识价值在于：

| 要提炼的内容 | 要跳过的内容 |
|------------|------------|
| 为什么要做这件事（动机/背景） | 批量生成的文案/代码/数据本身 |
| 架构设计是什么 | 验证用的示例输出 |
| 中间做了哪些调整，以及为什么调整 | 重复结构的执行结果 |
| 最终实现方案是什么 | |
| 踩了哪些坑，怎么解决的 | |

提炼时完全忽略批量输出内容，专注还原「设计 → 决策 → 迭代 → 结论」这条主线。

---

### Step 3：过滤对话中的其他噪音

去除批量任务之外，对话中还存在其他不需要进入知识文档的内容：

| 噪音类型 | 判断标准 | 处理方式 |
|---------|---------|---------|
| 被推翻的思路 | 后续对话中被否定、放弃或替换的方向 | 跳过，不记录 |
| 来回拉锯 | 反复确认同一件事，没有产生新信息 | 只保留最终结论 |
| 中间修改过程 | 同一内容改了多版，只有最终版成立 | 只提炼最终版及其原因 |
| 无效调试 | 报错信息、走不通的尝试、被放弃的方案 | 跳过，除非「为什么走不通」本身有复用价值（则记入踩坑记录） |

**唯一例外**：背景铺垫（为什么做这件事、出发点、目标）——无论出现在对话哪个位置，**必须保留**，这是知识文档最重要的上下文。

**核心原则**：只提炼「最终成立的结论」和「让这个结论成立的来龙去脉」，过程中被推翻的一切都是噪音。

---

### Step 4：检测对话是否需要拆分为多份文档

**在判断内容类型之前，先做这一步。** 一段对话可能包含多个独立的知识单元，强行合并会导致文档主题模糊、知识颗粒度丢失。

判断是否需要拆分的信号：

| 信号 | 示例 |
|------|------|
| 话题发生明显跳转 | 从「搭建某个工具」转向「如何分发/推广这个工具」 |
| 对话存在明显的阶段边界 | 「做完之后」「接下来」「那另一个问题是」 |
| 两段内容的目标读者不同 | 一段是「如何做」，另一段是「如何用/推广」 |
| 两段内容放进不同知识库分类更合适 | 一段属于「工具构建」，另一段属于「工具使用指南」 |

**处理规则**：
- 检测到明显分割点 → 拆成多份文档，分别生成，每份文档独立完整
- 内容虽多样但围绕同一核心目标 → 合并为一份，用清晰的章节结构区分

拆分时告知用户：「我检测到这段对话有 X 个独立主题，将生成 X 份文档：[列出各文档主题]」

---

### Step 5：分析每段对话，判断内容类型

对每一份待生成的文档，分别判断其内容类型（可复合）：

| 类型 | 识别信号 |
|------|---------|
| 💡 观点/灵感 | 讨论想法、趋势、洞察、金句 |
| 📚 方法论 | 步骤、框架、流程、最佳实践 |
| 🛠️ 项目/工作流 | 搭建过程、架构设计、工具链 |
| 💻 代码/技术 | 包含代码块、调试过程、技术实现 |
| 🔍 案例分析 | 具体产品、工具、行业案例研究 |

类型决定文档使用哪个模板（见下方）。

---

### Step 6：生成结构化 Markdown 文档

根据内容类型，选择对应模板生成文档。若有多份，依次生成后一并呈现给用户。

---

## 文档模板

### 通用文档头部（所有类型都有）

```markdown
---
title: {标题}
date: {今天日期，格式 YYYY-MM-DD}
tags: [{标签1}, {标签2}, ...]
type: {内容类型，如：方法论 / 项目笔记 / 技术实现 / 灵感观点 / 案例分析}
source: Claude 对话提炼
---

# {标题}

> **一句话摘要**：用一句话说清楚这篇文档在讲什么，以及它的核心价值是什么。
```

---

### 模板 A：观点/灵感/方法论类

适用：讨论类、方法论、思想框架、AI 内容创作心得等。

```markdown
## 核心观点

{3-5 条提炼好的核心观点，每条都是一个完整、独立、可直接引用的句子}

---

## 详细展开

### {观点1 标题}
{对这个观点的展开说明，结合对话内容提炼，不是复述原文}

**关键原句**：
> "{从对话中保留的 1-2 句最能代表这个观点的原始表达}"

### {观点2 标题}
...（同上）

---

## 关联知识节点

{识别这段对话涉及到的更大知识框架，列出 3-5 个知识标签/概念节点}

例如：
- **知识领域**：Prompt Engineering / 内容运营 / AI 工作流自动化
- **相关概念**：{概念1} → {概念2} → {概念3}（展示知识之间的关联）
- **可参考资源**：{如果对话中提到了工具/方法，列出它们的名称方便后续查找}

---

## 延伸思考

{基于这段对话，提出 2-4 个值得进一步探索的问题或方向}

- ❓ {问题1：这个方法是否可以应用到...？}
- ❓ {问题2：如果...会发生什么？}
- 🔮 {未来方向：这个方向可能演变成...}

---

## 可用素材

{如果对话中有直接可以用作内容选题、金句、案例的材料，在这里单独列出}

- 📌 选题方向：{...}
- 💬 金句："{...}"
- 📊 数据/案例：{...}
```

---

### 模板 B：项目/工作流/代码类

适用：包含代码、架构设计、工作流搭建、工具链配置等。

```markdown
## 项目概述

- **目标**：{这个项目/工作流要解决什么问题}
- **最终成果**：{实现了什么，或者达到了什么状态}
- **技术栈/工具**：{涉及的工具、语言、平台}

---

## 实现过程

### 阶段1：{阶段名称}
{这个阶段做了什么，解决了什么问题}

**关键决策**：
> "{保留 1 句最能代表这个阶段核心决策的原句}"

### 阶段2：{阶段名称}
...（同上）

---

## 核心代码/配置

{如果对话中有关键代码，在这里保留最核心的片段，并附上说明}

```{语言}
{代码}
```

**说明**：{这段代码的作用和注意事项}

---

## 踩坑记录

{对话中遇到的问题和解决方案，格式：问题 → 原因 → 解决方法}

| 问题 | 原因 | 解决方法 |
|------|------|---------|
| {问题1} | {原因} | {解决方法} |

---

## 可复用的模式

{从这次项目中提炼出来的、下次可以直接复用的方法/模板/思路}

---

## 关联知识节点

{同模板 A}

---

## 延伸思考

{同模板 A}
```

## 输出规范

1. **语言自适应**：检测对话使用的主要语言，文档框架、标题、模板标签全部使用同一语言输出。中文对话 → 中文文档，英文对话 → 英文文档，混合对话 → 以用户发言为准。
2. **文件命名**：`YYYY-MM-DD_{标题}.md`，标题使用对话的主要语言，不加特殊符号
2. **生成文件后**：用 `present_files` 工具把文件呈现给用户下载
3. **文件存放位置**：`/mnt/user-data/outputs/`
4. **代码处理原则**：
   - 对话中的代码不要全部保留，只保留**核心逻辑片段**
   - 大段调试过程提炼成"踩坑记录"，不原样复制
5. **提炼原则**：
   - 用你自己的语言重新组织，不是复述对话
   - 关键原句用 `>` blockquote 格式保留，每个观点最多 1-2 句
   - 所有结论都要有"为什么"，不只写结果

---

## 质量检查清单

生成文档后，自我检查：
- [ ] **是否漏判了拆分？** 这份文档有没有把两个本该独立的主题强行合并？
- [ ] 一句话摘要是否清晰表达了文档价值？
- [ ] 核心观点是否可以脱离上下文独立理解？
- [ ] 关联知识节点是否帮助这篇文档"接入"更大的知识体系？
- [ ] 延伸思考是否有实际探索价值，而不是废话？
- [ ] 如果是项目类，踩坑记录是否足够具体可复用？
